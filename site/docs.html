LCP v1 Specification

Complete reference for the Library Context Protocol v1.0

Overview

The Library Context Protocol (LCP) is a language-agnostic, machine-readable format for describing software libraries in a way that is optimized for AI systems, tooling, and documentation generators.

An LCP document captures:

  • What a library is (metadata)
  • What symbols it exposes (functions, classes, modules, etc.)
  • How those symbols behave (semantics, signatures, effects, stability)
  • How symbols are identified in a stable, canonical way

The primary design goal is to provide high-fidelity context about libraries so that LLMs, IDEs, and analysis tools can reason about APIs accurately without relying on source code parsing at runtime.

Design Principles

LCP is built around a few core principles:

  • Symbol-centric — Everything revolves around well-defined symbols
  • Stable identifiers — Symbol IDs should not change across refactors
  • Language-agnostic — Works across Python, JS/TS, Java, C#, Rust, etc.
  • Tool-friendly — Easy to diff, index, validate, and generate
  • Extensible — Vendor or ecosystem-specific data via x-* fields

Document Structure

An LCP file is a single JSON document with three main top-level sections:

json 
{
  "manifest": { ... },
  "symbols": { ... },
  "deprecations": { ... },
  "detailed_index": { ... }
}

Only manifest and symbols are required.

Manifest

The manifest section describes the library as a whole.

Required Fields

json 
"manifest": {
  "schema_version": "1.0",
  "library": {
    "name": "example-lib",
    "version": "1.2.3",
    "language": "python"
  }
}
  • schema_version — LCP schema version (currently 1.0)
  • library.name — Canonical library name
  • library.version — Semantic version (MAJOR.MINOR.PATCH)
  • library.language — Primary implementation language

Optional Metadata

The manifest may also include:

  • runtime_language — Runtime language if different (e.g., Python bindings over C++)
  • bindings — Binding type or target ecosystem
  • compatibility — Runtime or platform compatibility
  • distribution — Package registry (pypi, npm, cargo, etc.)
  • license — SPDX-compatible identifier or free text
  • changelog — URL and format
  • generation — Tooling metadata (generator name, version, date)

Symbol Resolution

"symbol_resolution": "fully-qualified"

Controls how symbols are identified. The recommended and default mode is fully-qualified.

Symbols

The symbols object is the core of LCP. Each key is a Symbol ID, and each value is a symbol description.

json 
"symbols": {
  "math:sin": { ... },
  "pathlib:Path": { ... }
}

Symbol Kinds

Each symbol has a kind, one of:

  • module
  • function
  • class
  • method
  • attribute
  • constant

Minimal Symbol Shape

json 
"math:sin": {
  "kind": "function",
  "semantics": {
    "summary": "Return the sine of x (measured in radians)."
  }
}

Only kind and semantics.summary are required.

Symbol Identification

LCP uses stable, fully-qualified Symbol IDs.

Canonical Format

<module_path>:<entity_path>

Examples:

  • json:loads
  • collections:Counter
  • collections:Counter#update
  • System.IO:File#ReadAllText

Modules as Symbols

Modules are represented using an empty entity path:

collections:

This allows module-level documentation and grouping.

Members and Nesting

  • Class member: module:Class#method
  • Nested type: module:Outer.Inner
  • Nested member: module:Outer.Inner#method

Overloads

Overloaded functions or methods must not use different Symbol IDs. Instead, multiple entries go into signatures[] under the same symbol.

Signatures

signatures describe callable forms of functions or methods.

json 
"signatures": [
  {
    "params": [
      { "name": "path", "type": "string", "required": true }
    ],
    "returns": "string"
  }
]

Each signature may define:

  • when — Conditional applicability
  • async — Whether the call is asynchronous
  • params — Parameter list
  • returns — Return type
  • raises — Errors or exceptions

Parameter Properties

Property Type Description
name string Parameter name (required)
type type_ref Type annotation (required)
required boolean Whether parameter is required (default: true)
default any Default value if not required
variadic boolean Whether this is a variadic parameter
kind string positional, keyword, positional_only, keyword_only, rest
description string Parameter documentation

Types

Types are represented using type_ref.

Simple Types

"returns": "string"

Structured Types

json 
{
  "kind": "array",
  "items": "string"
}

Supported kinds:

  • named — Named type reference
  • array — Array/list of items
  • map — Key-value mapping
  • tuple — Fixed-length tuple
  • union — Union of multiple types

This allows lossless representation of complex type systems.

Semantics

semantics describe what a symbol does, independent of how it is implemented.

json 
"semantics": {
  "summary": "Reads a file and returns its contents.",
  "description": "The file must exist and be readable.",
  "examples": [
    { "code": "read_file('a.txt')" }
  ]
}
  • summary is required and should be concise
  • description and examples are optional but strongly encouraged

Effects

The optional effects section describes observable behavior:

json 
"effects": {
  "categories": ["filesystem"],
  "idempotent": true,
  "deterministic": true
}

Supported effect categories:

  • io — General I/O operations
  • network — Network calls
  • filesystem — File system access
  • cpu — CPU-intensive operations
  • memory — Memory-intensive operations
  • gpu — GPU operations

This is particularly useful for AI reasoning and sandboxing.

Stability

Symbols may declare stability information:

json 
"stability": {
  "level": "stable",
  "since": "1.0.0"
}

Levels:

  • experimental — May change without notice
  • stable — Follows semver guarantees
  • deprecated — Scheduled for removal

Deprecations

Global deprecations can be declared separately:

json 
"deprecations": {
  "old.module:func": {
    "deprecated_in": "2.0.0",
    "replaced_by": "new.module:func"
  }
}

This allows tracking removals even when symbols are no longer present.

Detailed Index

detailed_index optionally links symbols back to source artifacts:

json 
"detailed_index": {
  "math:sin": {
    "implementation": {
      "path": "math.py",
      "lines": [10, 42]
    }
  }
}

This is useful for IDEs, documentation generators, and traceability.

Extensibility

Any object in LCP may include vendor extensions using x-* keys:

json 
"x-internal-id": "abc123"

These are ignored by the core schema but preserved by tooling.

Summary

LCP provides a compact, precise, and extensible way to describe libraries at the semantic level. By standardizing symbol identity and behavior, it enables tools—and AI systems in particular—to understand APIs as humans do: by intent, contract, and effect, not just syntax.
